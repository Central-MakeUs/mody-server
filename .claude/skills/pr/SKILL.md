# pr

mody-server의 Pull Request를 생성한다. **PR 본문은 반드시 해당 기능의 LLD 문서를 오라클(ground truth)로 사용해 작성한다.**

## ⚠️ 최우선 규칙 — 절대 위반 금지

1. **임의 판단 금지.** LLD / ADR / 실제 코드 diff에 명시되지 않은 내용을 추측으로 채우지 않는다.
2. 결정되지 않은 항목 — 특히 LLD의 `## 미결정 사항(Open Questions)` 에 있는 것 — 은 PR 본문에서 **빈칸 + `> ⚠️ 미결정 — 확인 필요`** 로만 남긴다.
3. 작성 후, 빈칸으로 남긴 항목을 사용자에게 보고하고 확인받는다.
4. 오라클로 쓸 LLD를 못 찾으면 **진행을 멈추고** 어떤 LLD가 기준인지 사용자에게 묻는다. LLD 없이 본문을 지어내지 않는다.

## 사전 조건 (TBD / Trunk-Based)

- base 브랜치는 **항상 `main`**. (main 직머지 → 머지 시 자동 배포)
- 작업 브랜치는 `feature/{issue-number}` 형식. 현재 브랜치가 다르면 사용자에게 확인.
- main은 머지 즉시 배포되므로, PR은 작고 안전하게. 미완성/실패 테스트 포함 시 본문에 명시.

## 절차

1. 현재 브랜치와 `git diff main...HEAD`로 변경 내용을 파악한다. 브랜치명이 `feature/{issue-number}`인지 확인한다.
2. 오라클 LLD를 찾는다: `docs/lld/LLD-*.md` 중 이 기능/이슈에 해당하는 문서. 모호하거나 없으면 **멈추고 묻는다.**
3. LLD 전체와 거기서 참조하는 ADR(`docs/adr/ADR-*.md`)을 읽는다.
4. 아래 템플릿으로 본문을 작성한다. 출처는 오직 (a) LLD, (b) 코드 diff, (c) 참조된 ADR.
5. `gh pr create --base main` 으로 생성한다. 제목은 커밋 컨벤션과 동일하게 `<type>(<scope>): <영문 제목>`.
6. 빈칸으로 남긴 미결정 항목을 사용자에게 요약 보고한다.

## PR 본문 템플릿

```markdown
## 개요
<LLD의 목적/배경 요약 — LLD에 적힌 내용만>

## 관련 문서
- Issue: Closes #<N>   <!-- 이슈를 완전히 해소할 때만 Closes, 아니면 Refs -->
- LLD: docs/lld/LLD-XXXX-<slug>.md  (오라클)
- ADR: docs/adr/ADR-XXXX-<slug>.md  <!-- 있을 때만 -->

## 변경 사항
<코드 diff 기반 사실만 나열>

## 구현 ↔ LLD 대조
| LLD 항목 | 반영 | 비고 |
| --- | --- | --- |
| <범위/인터페이스/데이터모델 등> | ✅ / 부분 / ❌ | |

## 인수조건 체크리스트
<!-- LLD 7장 인수조건을 그대로 -->
- [ ] <조건>

## 미결정 사항 (확인 필요)
> ⚠️ 아래는 LLD에서 결정되지 않은 항목입니다. 작성자 확인 전까지 비워둡니다.
- <LLD Open Questions 항목 그대로. 추측으로 답을 채우지 말 것>

## 테스트
<실행한 테스트와 결과. 안 했으면 "미실행"이라고 정직하게>
```

## 하지 말 것

- LLD에 없는 설계/결정을 본문에 서술하기.
- 미결정 항목을 그럴듯하게 채우기.
- 인수조건을 임의로 만들어 넣기 (LLD에서만 가져온다).
- 테스트를 안 했는데 통과했다고 쓰기.
- base를 main 외 다른 브랜치로 잡기 (확인 없이).
