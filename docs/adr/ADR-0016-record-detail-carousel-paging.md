# ADR-0016: 기록 상세 캐러셀 API 페이징 전략

## 1. 상태
Accepted

## 2. 날짜
2026-07-04

## 3. 컨텍스트 (Context)
* 기존 `GET /api/v1/records/{recordId}` API는 유저의 당일 기록 상세 목록(식사/운동)을 캐러셀 뷰용으로 조회할 때, 하루 분량의 데이터가 작을 것으로 판단해 전체 일괄 조회 방식을 취했습니다.
* 하지만 클라이언트에게 페이지 크기 조작 권한(API 선택지)을 부여하고, 혹여나 하루에 대량의 데이터를 등록하는 엣지 케이스나 미래의 캐러셀 스크롤 확장성에 대응하기 위해 **커서 기반 페이징(Cursor-based Paging)**을 도입하기로 결정했습니다.

## 4. 의사결정 (Decision)
* **상세 조회 API 페이징 매개변수 추가**:
  * `GET /api/v1/records/{recordId}` API에 `cursor` (Long, optional)와 `size` (int, default=20) 쿼리 파라미터를 추가적으로 수용합니다.
* **타겟 기록 우선 반환 전략 (Target Record Centered)**:
  * 커서 페이징을 순차 적용할 경우, 최초 상세 조회 시(`cursor == null`) 사용자가 클릭해 진입한 타겟 기록(`recordId`)이 첫 번째 페이지에 반드시 포함되어야 합니다.
  * 이를 위해 `cursor`가 비어 넘어올 때는 내부 쿼리의 조회 커서값을 **`recordId - 1`**로 보정 설정합니다.
  * 이로 인해 오름차순 페이징 결과의 첫 번째 데이터(`index 0`)가 타겟 기록이 되며, 이후 캐러셀 슬라이딩 시 반환되는 `nextCursor`로 뒤의 데이터들을 추가 페이징해올 수 있습니다.
* **전체 통계 제공**:
  * 당일 작성자가 남긴 전체 기록 건수는 별도의 `COUNT` 쿼리를 호출하여 응답의 `totalCount`로 항시 유지해 줍니다.

## 5. 결과 (Consequences)
* 클라이언트 캐러셀 뷰에서 대용량 데이터 로드가 필요할 시 점진적으로 스크롤링 페이징할 수 있는 유연한 스펙이 확립되었습니다.
* 첫 페이지 로딩 시 항상 진입한 글이 배열의 첫머리에 오도록 보장되어 모바일 캐러셀 구현이 간결해집니다.
