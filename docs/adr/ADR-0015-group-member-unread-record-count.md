# ADR-0015: 그룹원별 미확인 기록 수 계산 전략

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-04 |
| 관련 이슈 | #64 |

## 1. 맥락

그룹 페이지의 구성원 목록은 현재 로그인 회원이 각 그룹원의 기록을 몇 개 아직 확인하지 않았는지 응답해야 한다.
기준은 “그룹원의 기록 상세 페이지로 넘어간 이후 새로 올라온 기록 수”다.

이 값은 그룹원 목록 화면에서 반복적으로 노출되므로 계산 기준과 저장 모델을 명확히 정해야 한다.

## 2. 대안

### 대안 A: 그룹원별 마지막 상세 진입 시각 저장

- 방식: `viewer_member_id`, `group_id`, `writer_member_id`, `last_viewed_at`을 저장한다.
- 그룹원 조회 시 `last_viewed_at` 이후 작성된 활성 기록 수를 계산한다.
- 장점: 저장 row 수가 작고, 상세 진입 1회로 해당 작성자의 기존 기록을 모두 확인 처리할 수 있다.
- 단점: 특정 기록 단위의 읽음/안읽음 상태는 표현할 수 없다.
- 적합한 상황: 현재 화면 요구처럼 “그 사람 게시글을 몇 개 안 봤는지”만 필요할 때.

### 대안 B: 기록별 읽음 테이블 저장

- 방식: `record_id`, `viewer_member_id`, `read_at`을 저장한다.
- 장점: 기록 단위 읽음 상태를 정확히 표현할 수 있다.
- 단점: 기록 수와 조회자 수에 비례해 row가 크게 증가한다.
- 보류 이유: 현재 요구는 작성자 단위 카운트라 기록별 읽음까지는 과하다.

### 대안 C: 클라이언트 로컬 기준 저장

- 방식: 앱이 마지막 확인 시각이나 마지막 record id를 로컬에 저장하고 서버는 전체 기록 수만 응답한다.
- 장점: 서버 저장소가 단순하다.
- 단점: 기기 변경/재설치/로그아웃 후 상태가 사라지고, 여러 기기에서 일관성이 깨진다.
- 보류 이유: 그룹 피드 상태는 서버 기준으로 일관되게 제공하는 편이 안전하다.

## 3. 결정

1차 구현은 **그룹원별 마지막 상세 진입 시각 저장**을 채택한다.

- `record_view_history` 테이블을 추가한다.
- 기록 상세 진입 시 그룹 기록이면 해당 작성자에 대한 `last_viewed_at`을 갱신한다.
- 그룹원 조회 시 `last_viewed_at` 이후 올라온 활성 기록 수를 `unreadRecordCount`로 응답한다.
- 본인 행은 읽음 배지 대상이 아니므로 `0`으로 응답한다.

## 4. 결과

긍정적 효과:

- 서버 기준으로 기기 간 일관된 미확인 기록 수를 제공할 수 있다.
- 저장 row 수가 `조회자 x 그룹원` 수준으로 제한된다.
- 상세 진입 한 번으로 해당 작성자의 기존 기록을 모두 확인 처리하는 UX와 맞다.

부정적 효과 / 리스크:

- 특정 기록을 개별적으로 읽음 처리하는 UX가 생기면 테이블 구조를 확장해야 한다.
- 상세 진입 시점 갱신이 실패하면 그룹원 목록의 카운트가 줄지 않을 수 있다.
- 그룹 탈퇴/소프트 삭제 정책과 함께 조회 조건을 유지해야 한다.

## 5. 재검토 기준

- 기록별 읽음 표시가 필요해질 때.
- “상세 진입”이 아니라 “캐러셀에서 특정 기록까지 넘긴 시점”을 기준으로 삼아야 할 때.
- 그룹원 수와 조회 패턴 증가로 그룹원 목록 조회가 느려질 때.
