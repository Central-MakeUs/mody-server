# ADR-0004: 엔티티와 도메인 모델 통합

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-06-29 |
| 관련 | ADR-0003 |

## 맥락 (Context)

모디는 Spring Boot, Spring Data JPA, MySQL을 중심으로 빠르게 API를 확장하는 초기 단계의 백엔드다.
이 단계에서 순수 도메인 모델과 JPA 엔티티를 완전히 분리하면 다음 비용이 발생한다.

- `Controller DTO`와 도메인 모델, JPA 엔티티 사이의 변환 로직이 반복된다.
- 단순 CRUD와 조회 API에도 매퍼, 도메인 객체, 엔티티 클래스가 함께 늘어난다.
- 기능 변경 시 필드 하나를 여러 계층에 반영해야 하므로 개발 속도와 리뷰 비용이 증가한다.
- 현재 저장소 기술은 RDB와 JPA가 명확하며 단기간에 다른 저장소로 전환할 가능성이 낮다.

반대로 엔티티를 단순 테이블 매핑 객체로만 두면 서비스 계층에 검증, 상태 전이, 불변식 로직이
흩어질 위험이 있다.

## 결정 (Decision)

메인 도메인 모델과 JPA 엔티티를 하나의 클래스로 통합한다.
JPA 의존성은 현재 기술 선택에 따른 허용 가능한 비용으로 본다.

세부 기준은 다음과 같다.

- 비즈니스 규칙을 가진 주요 모델에 `@Entity`를 직접 적용한다.
- `jakarta.persistence` 의존은 표준 영속성 스펙 의존으로 수용한다.
- API 요청/응답 DTO는 엔티티와 분리한다.
- 값 자체에 규칙이 있는 필드는 `@Embeddable` 또는 VO로 분리한다.
- 생성 검증, 상태 전이, 불변식 검증은 가능한 엔티티/VO 메서드에 둔다.
- 서비스 계층은 트랜잭션 경계, 저장소 조합, 외부 연동 조정에 집중한다.

## 고려한 대안 (Considered Options)

1. **도메인 모델과 JPA 엔티티 완전 분리** — 저장소 독립성이 높지만 매핑 비용과 클래스 수가 증가한다.
2. **JPA 엔티티를 테이블 매핑 전용 객체로 사용** — 구현은 단순하지만 비즈니스 로직이 서비스로 누수된다.
3. **엔티티와 도메인 모델 통합** — JPA에 결합되지만 초기 제품 개발 속도와 응집도를 확보할 수 있다.

## 결과 (Consequences)

### 긍정

- 매핑 보일러플레이트와 클래스 수를 줄인다.
- 기능 흐름을 DTO, 서비스, 엔티티, 저장소 수준에서 빠르게 추적할 수 있다.
- 엔티티가 상태와 행위를 함께 가지도록 유도한다.
- VO를 통해 복잡한 값 규칙을 독립적으로 테스트할 수 있다.

### 부정 / 트레이드오프

- 저장소 기술을 바꿀 경우 엔티티 리팩토링 비용이 발생한다.
- 일부 테스트는 JPA 동작 방식을 고려해야 한다.
- 엔티티가 커지면 책임이 과도하게 모일 수 있으므로 VO와 도메인 메서드 분리가 필요하다.

## 후속 / 미결정

- 도메인별 VO 도입 기준을 코드 리뷰에서 계속 점검한다.
- 엔티티가 단순 setter 중심으로 변질되지 않도록 생성/변경 메서드 규칙을 유지한다.
