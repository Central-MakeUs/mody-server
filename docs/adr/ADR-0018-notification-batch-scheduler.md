# ADR-0018: 알림 배치와 스케줄러 운영 전략

## 상태

Accepted

## 날짜

2026-07-04

## 맥락

알림 발송은 ADR-0012에 따라 notification row를 outbox로 사용한다.
이 구조가 실제 서비스에서 동작하려면 발송 대상 선점, 식사/운동 리마인더 생성,
그룹 연속 기록 위험 알림 생성, 오래된 알림 정리 같은 배치성 작업이 필요하다.

이 작업들을 API 요청 흐름에 넣으면 응답 시간이 길어지고 외부 FCM 상태가 사용자 요청 트랜잭션에 섞인다.
반대로 별도 큐나 외부 스케줄러 플랫폼을 바로 도입하면 현재 배포 구조 대비 운영 복잡도가 커진다.

## 결정

알림 배치성 작업은 **Spring Scheduler 기반 애플리케이션 내부 스케줄러**로 시작한다.

- 모든 스케줄러는 환경변수로 켜고 끌 수 있게 하며 기본값은 `false`로 둔다.
- 발송 배치는 `notification.dispatch.*` 설정을 사용한다.
- 식사/운동 리마인더 생성은 `notification.reminder.*` 설정을 사용한다.
- 그룹 연속 기록 위험 알림 생성은 `notification.streak-risk.*` 설정을 사용한다.
- 알림 보관 기간 정리는 `notification.retention.*` 설정을 사용한다.
- 중복 생성은 notification row의 `dedupeKey`로 방지한다.
- 다중 인스턴스 발송 중복은 `FOR UPDATE SKIP LOCKED`로 due 알림을 선점한 뒤
  `PROCESSING`, `pickedBy`, `pickedAt` 상태를 저장해 방지한다.
- FCM token 단위 응답에서 `UNREGISTERED` 또는 token 단위 `INVALID_ARGUMENT`가 오면 해당 token을 비활성화한다.
- 운영 부하가 커지면 외부 큐/워커 또는 외부 스케줄러로 분리한다.

## 고려한 대안

1. **API 요청 시점에 동기 처리**
   - 장점: 별도 스케줄러가 필요 없다.
   - 단점: 사용자 응답 지연과 외부 시스템 장애 전파가 커진다.

2. **Spring Scheduler 사용**
   - 장점: 현재 배포 구조에서 추가 인프라 없이 구현할 수 있고 설정이 단순하다.
   - 단점: 다중 인스턴스 운영 시 중복 실행 가능성을 코드와 DB 상태로 제어해야 한다.

3. **외부 큐와 워커 도입**
   - 장점: 발송량 증가, 재시도, 모니터링에 유리하다.
   - 단점: 지금 단계에서는 운영 구성과 장애 지점이 늘어난다.

4. **Cloud Scheduler/Batch 등 외부 스케줄러 사용**
   - 장점: 스케줄 실행 주체를 애플리케이션 밖으로 분리할 수 있다.
   - 단점: 로컬/개발 환경 재현성이 떨어지고 배포 설정이 추가된다.

## 결과

### 긍정

- 알림 발송과 생성 배치를 API 응답 흐름에서 분리할 수 있다.
- 배치별 환경변수로 dev/prod에서 점진적으로 켤 수 있다.
- dedupe key와 선점 상태로 중복 생성/중복 발송을 제어할 수 있다.
- 여러 워커가 동시에 실행되어도 이미 잠긴 알림 row를 기다리지 않고 건너뛰므로 락 대기를 줄일 수 있다.
- FCM invalid token을 즉시 비활성화해 불필요한 재발송을 줄일 수 있다.
- 현재 단일 애플리케이션 배포 구조와 맞아 초기 운영 부담이 낮다.

### 부정 / 트레이드오프

- 스케줄러가 애플리케이션 내부에 있어 서버 인스턴스 수가 늘면 중복 실행을 계속 고려해야 한다.
- 배치 실행 이력과 실패 모니터링은 별도 관리 기능이 없으면 로그에 의존한다.
- 대량 알림 발송이 필요해지면 애플리케이션 스레드풀과 DB 부하를 다시 조정해야 한다.
- quiet time, 야간 발송 제한 같은 사용자 경험 정책은 별도 결정이 필요하다.

## 후속 결정

- 야간 발송 제한 또는 quiet time 정책.
- 배치 실행 이력 테이블이 필요한 시점.
- 외부 큐/워커로 분리할 임계치.
