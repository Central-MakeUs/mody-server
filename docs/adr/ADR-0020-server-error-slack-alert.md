# ADR-0020: 500 에러 Slack 알림 전략

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-12 |
| 관련 이슈 | #92 |

## 1. 맥락

운영 중 예상치 못한 500 계열 오류가 발생하면 서버 로그를 직접 확인하기 전까지 인지하기 어렵다.
초기 운영 단계에서는 별도 APM보다 팀이 바로 확인할 수 있는 Slack 알림이 실용적이다.

다만 오류 알림이 API 응답 흐름을 지연시키거나, Slack 장애로 서버 응답까지 실패하면 안 된다.
또한 요청 body에는 민감 정보가 포함될 수 있으므로 알림 대상에서 제외한다.

## 2. 결정

- 500 계열 서버 오류는 전역 예외 처리에서 Slack 알림을 요청한다.
- Slack 전송은 비동기로 수행해 API 응답 경로와 분리한다.
- Slack webhook URL은 환경 변수로 주입하고, 기본값은 비활성화한다.
- 알림에는 요청자 정보와 요청 메타데이터만 포함한다.
  - 가능한 경우 `memberId`, `nickname`
  - method, URI, query, client IP, User-Agent
  - exception class, message, 일부 stack trace
- 요청 body와 Authorization 헤더 값은 Slack에 남기지 않는다.
- Slack 전송 실패는 로그만 남기고 API 응답에 영향을 주지 않는다.

## 3. 결과

긍정적 효과:

- 운영 500 오류를 팀이 빠르게 인지할 수 있다.
- 오류가 발생한 요청자와 요청 경로를 함께 확인할 수 있다.
- Slack 장애나 webhook 설정 누락이 API 응답을 깨뜨리지 않는다.

부정적 효과 / 리스크:

- Slack webhook만으로는 집계, 검색, 장기 보관 기능이 제한적이다.
- 인증 전 실패한 요청은 회원 정보를 알 수 없어 `unknown`으로 남는다.
- 요청 body를 남기지 않기 때문에 일부 재현 정보는 로그와 함께 확인해야 한다.

## 4. 재검토 기준

- 운영 트래픽이 증가해 500 알림이 과도하게 발생할 때.
- APM 또는 로그 수집 시스템을 도입해 알림 채널을 통합할 때.
- 특정 오류에 대한 rate limit, dedupe, 알림 등급 분리가 필요해질 때.
