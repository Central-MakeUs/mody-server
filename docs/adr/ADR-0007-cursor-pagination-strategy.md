# ADR-0007: 커서 기반 페이징 전략

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-06-29 |
| 관련 | 그룹 피드, 기록 목록, 알림 목록 |

## 맥락 (Context)

모디의 주요 목록은 모바일 클라이언트에서 무한 스크롤로 소비된다.
식사/운동 기록, 알림, 챌린지 인증 이미지처럼 시간이 지날수록 데이터가 계속 쌓이는 목록은
offset 기반 페이징을 사용하면 뒤 페이지 조회 비용이 커지고 중간 삽입/삭제 시 중복 또는 누락이 생길 수 있다.

다만 관리자성 목록이나 고정된 코드 목록처럼 데이터 양이 작고 페이지 번호가 필요한 화면은
offset 방식이 더 단순할 수 있다.

## 결정 (Decision)

사용자-facing 무한 스크롤 목록은 커서 기반 페이징을 기본으로 한다.

- 기본 정렬은 최신순이며 `createdAt`과 `id`를 함께 사용해 안정적인 순서를 만든다.
- 같은 시각 데이터가 있어도 `id`를 tie-breaker로 사용한다.
- 응답은 `items`, `nextCursor`, `hasNext` 형태를 기본으로 한다.
- 총 개수는 기본 응답에 포함하지 않는다.
- `size`는 서버에서 최대값을 제한한다.
- 데이터 양이 작고 변경이 적은 목록은 offset 페이징을 허용한다.

## 고려한 대안 (Considered Options)

1. **offset 페이징 일괄 적용** — 구현은 쉽지만 대량 데이터와 실시간 삽입에 취약하다.
2. **커서 페이징 일괄 적용** — 모바일 목록에 적합하지만 페이지 번호 이동에는 부적합하다.
3. **화면 성격별 선택** — 기준 관리가 필요하지만 성능과 구현 단순성을 균형 있게 가져간다.

## 결과 (Consequences)

### 긍정

- 무한 스크롤에서 중복/누락 가능성을 줄인다.
- 뒤 페이지 조회 시 DB 스캔 비용을 낮출 수 있다.
- 클라이언트는 `nextCursor`만 보관하면 다음 목록을 요청할 수 있다.

### 부정 / 트레이드오프

- 특정 페이지 번호로 바로 이동하기 어렵다.
- 커서 인코딩/디코딩과 정렬 조건을 API별로 일관되게 관리해야 한다.
- 필터 조건이 복잡해지면 커서 구성도 함께 확장해야 한다.

## 후속 / 미결정

- 커서 문자열 포맷은 API 구현 시 공통 유틸로 고정한다.
- 정렬 기준이 날짜가 아닌 목록은 별도 ADR 또는 LLD에서 커서 필드를 명시한다.
