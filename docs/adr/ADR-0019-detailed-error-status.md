# ADR-0019: 클라이언트 친화적 세부 에러 코드 전략

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-11 |
| 관련 이슈 | #85 |

## 1. 맥락

API 실패 응답은 `isSuccess`, `code`, `message`, `result` 구조로 일관되게 제공하고 있다.
다만 입력값 오류가 `MEMBER301`, `GROUP301`, `RECORD301`처럼 넓은 코드로만 내려가면 클라이언트가 어떤 값을 고쳐야 하는지 즉시 판단하기 어렵다.

응답 구조를 확장하면 클라이언트 모델 변경이 필요하므로, 현재 구조를 유지하면서 `ErrorStatus` 자체를 더 세분화하는 전략이 필요하다.

## 2. 결정

- 기존 실패 응답 구조는 변경하지 않는다.
- 클라이언트가 `code`와 `message`만 보고 대응할 수 있도록 `ErrorStatus`를 정책 단위로 세분화한다.
- 세분화 단위는 개별 필드 하나하나가 아니라 화면 정책과 사용자 행동 기준으로 정한다.
  - 예: 운동 일정 요일/시간 누락, 식사 시간 조합 오류, 기록 이미지 키 오류.
- `@Valid` 실패는 전역 예외 처리에서 세부 `ErrorStatus`로 변환한다.
- 세부 매핑이 없는 검증 실패는 기존 도메인 대표 검증 오류 코드로 fallback한다.

## 3. 적용 원칙

- 응답 예시:

```json
{
  "isSuccess": false,
  "code": "MEMBER309",
  "message": "운동 일정은 요일과 시간을 함께 입력해야 합니다."
}
```

- 기존 클라이언트 호환성을 위해 추가 필드는 만들지 않는다.
- 서버 내부 디버깅용 상세 field error는 로그와 Swagger 명세로 보완한다.
- 1차 적용 범위는 클라이언트 테스트 중 혼동 가능성이 높은 온보딩 개인 정보 입력과 기록/댓글 입력으로 제한한다.

## 4. 결과

긍정적 효과:

- 클라이언트가 별도 응답 구조 변경 없이 더 정확한 원인을 알 수 있다.
- Swagger 예외 예시와 실제 런타임 응답이 더 가까워진다.
- 넓은 validation code에 의존하는 수동 커뮤니케이션을 줄일 수 있다.

부정적 효과 / 리스크:

- `ErrorStatus` 항목 수가 증가한다.
- validation 메시지와 status 매핑이 어긋나면 의도보다 넓은 fallback 코드가 내려갈 수 있다.
- 필드별로 과도하게 쪼개면 클라이언트 분기 부담이 커질 수 있으므로 정책 단위 분리를 유지해야 한다.

## 5. 재검토 기준

- 클라이언트가 field-level error 목록을 실제 UI에 활용해야 할 때.
- 동일한 에러 코드가 여러 화면에서 서로 다른 UX를 요구할 때.
- 세부 에러 코드 수가 많아져 관리가 어려워질 때.
