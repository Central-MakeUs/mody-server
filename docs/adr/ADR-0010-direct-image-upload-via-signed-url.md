# ADR-0010: Signed URL 기반 직접 이미지 업로드 채택

## 상태

Accepted

## 날짜

2026-06-29

## 맥락

모디는 식사/운동 기록, 프로필, 주간 챌린지 인증에서 이미지를 사용한다.
클라이언트가 이미지를 서버 API로 전송하고
서버가 다시 스토리지로 업로드하는 방식은 구현이 단순하지만,
이미지 바이너리가 애플리케이션 서버 메모리와 네트워크 대역폭을 크게 사용한다.

이미지 크기와 동시 업로드 수가 늘어나면 다음 문제가 발생할 수 있다.

- 서버 heap 사용량 증가와 GC 부담.
- 요청 처리 thread 점유 시간 증가.
- API 서버 outbound 트래픽 증가.
- 업로드 실패/재시도 시 서버 부하 집중.
- API 서버가 이미지 처리 병목이 되는 구조.

## 결정

이미지 업로드는 서버를 거치지 않고 클라이언트가 스토리지로 직접 업로드한다.
서버는 인증된 클라이언트에게 짧은 만료 시간을 가진 Signed URL과 image key만 발급한다.

1. 클라이언트가 업로드 목적과 파일명을 서버에 전달한다.
2. 서버가 image key를 생성하고 Signed URL을 발급한다.
3. 클라이언트가 Signed URL로 스토리지에 직접 업로드한다.
4. 클라이언트가 이후 기록/프로필/인증 API에 image key를 전달한다.

운영 환경의 스토리지는 Google Cloud Storage를 기본 대상으로 한다.
로컬/테스트 환경에서는 교체 가능한 issuer 구현으로 실제 GCS 의존 없이 검증한다.

## 고아 이미지

Signed URL 발급 후 클라이언트가 업로드만 하고 서버 도메인 API를 호출하지 않으면
DB에서 참조하지 않는 고아 이미지가 생길 수 있다.

고아 이미지는 실시간 트랜잭션에서 정리하지 않는다.
대신 스토리지 object 생성 시각과 DB 참조 여부를 기준으로 주기적 배치 작업에서 삭제한다.

초기 배치 정책:

- 일정 시간 이상 DB 참조가 없는 object를 삭제 대상으로 본다.
- record/profile/weekly-challenge 등 prefix별로 스캔 범위를 나눈다.
- 삭제 전 dry-run 로그를 남길 수 있게 한다.

## 대안

1. 서버 프록시 업로드
   - 장점: 클라이언트 구현이 단순하고 업로드 검증을 서버에서 즉시 수행할 수 있다.
   - 단점: 서버 메모리, thread, 네트워크 비용이 증가하고
     이미지 업로드가 API 서버 병목이 된다.

2. 클라이언트에 장기 스토리지 권한 부여
   - 장점: 서버 API 호출이 줄어든다.
   - 단점: 권한 범위가 넓어지고 보안 사고 영향 범위가 커진다.

3. Signed URL 직접 업로드
   - 장점: 서버 부하를 줄이고 권한 범위와 만료 시간을 좁게 유지할 수 있다.
   - 단점: 고아 이미지 정리 배치가 필요하다.

## 결과

- API 서버는 이미지 바이너리를 받지 않는다.
- 서버는 image key 생성과 Signed URL 발급만 담당한다.
- 실제 업로드 성공 여부와 도메인 데이터 연결은 후속 API 호출에서 확정된다.
- 고아 이미지는 별도 배치 작업으로 정리한다.
