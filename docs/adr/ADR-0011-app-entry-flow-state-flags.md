# ADR-0011: 앱 진입 플로우 상태 플래그 분리

## 상태

Accepted

## 날짜

2026-07-02

## 맥락

모디 앱은 로그인 이후 회원 상태에 따라 서로 다른 화면으로 진입해야 한다.
현재 클라이언트가 구분해야 하는 주요 상태는 다음과 같다.

- 개인정보 온보딩을 아직 완료하지 않은 상태.
- 개인정보는 완료했지만 그룹 온보딩을 한 번도 완료하지 않은 상태.
- 과거에는 그룹을 사용했지만 현재 활성 그룹이 없는 상태.
- 개인정보와 그룹 참여 상태가 모두 충족되어 메인으로 진입 가능한 상태.

단순히 현재 활성 그룹 존재 여부만 내려주면
최초 그룹 온보딩이 필요한 회원과,
기존 사용자가 모든 그룹을 나간 예외 케이스를 구분하기 어렵다.

## 결정

로그인 응답에는 앱 진입 분기를 위해 다음 플래그를 분리해서 제공한다.

- `personalInfoCompleted`: 개인정보 온보딩 완료 여부.
- `mainAccessible`: 현재 메인 화면에 진입 가능한지 여부.
- `groupOnboardingCompleted`: 그룹 생성 또는 참여를 통해 그룹 온보딩을 한 번이라도 완료한 이력.

`mainAccessible`은 현재 상태 기반 플래그다.
개인정보가 완료되어 있고, 사용 중인 활성 그룹이 하나 이상 있으면 `true`다.
활성 그룹이 하나도 없으면 과거 이력과 무관하게 `false`다.

`groupOnboardingCompleted`는 이력 기반 플래그다.
회원이 그룹 생성 또는 참여 플로우를 한 번이라도 완료하면 `true`가 되며,
이후 모든 그룹을 나가거나 그룹이 삭제되어도 다시 `false`로 되돌리지 않는다.

클라이언트 분기 우선순위는 다음과 같다.

1. 인증 정보가 없으면 로그인 화면으로 이동한다.
2. `personalInfoCompleted == false`면 개인정보 온보딩으로 이동한다.
3. `mainAccessible == true`면 메인 화면으로 이동한다.
4. `mainAccessible == false && groupOnboardingCompleted == false`면 최초 그룹 온보딩으로 이동한다.
5. `mainAccessible == false && groupOnboardingCompleted == true`면 그룹 재참여 또는 재생성 플로우로 이동한다.

## 고려한 대안

1. **`mainAccessible`만 제공**
   - 장점: 응답 필드가 적고 구현이 단순하다.
   - 단점: 최초 그룹 온보딩과 그룹을 모두 나간 기존 사용자를 구분할 수 없다.

2. **현재 활성 그룹 수를 직접 제공**
   - 장점: 클라이언트가 세부 상태를 직접 판단할 수 있다.
   - 단점: 앱 진입 정책이 클라이언트로 누수되고, 서버 정책 변경 시 클라이언트 수정 범위가 커진다.

3. **현재 상태 플래그와 이력 플래그 분리**
   - 장점: 서버가 진입 정책에 필요한 최소 의미를 명확히 제공하고 예외 케이스를 구분할 수 있다.
   - 단점: 그룹 온보딩 완료 이력을 별도 저장하고 유지해야 한다.

## 결과

- 클라이언트는 로그인 직후 일관된 조건으로 진입 화면을 결정할 수 있다.
- 최초 가입자와 기존 사용자의 그룹 없음 상태를 구분할 수 있다.
- 그룹 탈퇴 또는 삭제로 활성 그룹이 0개가 되어도 과거 그룹 온보딩 이력은 보존된다.
- 향후 그룹 재참여, 그룹 생성 안내 문구를 상태별로 다르게 제공할 수 있다.

## 주의 사항

- `groupOnboardingCompleted`는 "그룹을 생성한 적이 있는가"가 아니라
  "그룹 생성 또는 참여를 통해 그룹 온보딩을 완료한 적이 있는가"를 의미한다.
- 메인 진입 가능 여부는 반드시 `mainAccessible`을 기준으로 판단한다.
  `groupOnboardingCompleted`가 `true`여도 활성 그룹이 없으면 메인으로 이동하지 않는다.
