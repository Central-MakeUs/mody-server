# ADR-0013: API 상태 소유권 분리

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-04 |
| 관련 이슈 | #64 |

## 1. 맥락

온보딩, 마이페이지, 피드, 그룹 API가 늘어나면서 같은 상태를 여러 API에서 동시에 수정하는 구조가 생겼다.
특히 식사 시간은 개인정보 입력 API와 알림 설정 API 양쪽에서 받을 수 있었고,
식사/운동 알림 수신 여부는 클라이언트 화면에서는 하나의 스위치인데 서버 API는 두 필드로 나뉘어 있었다.

이 구조는 클라이언트가 어느 API를 원천으로 삼아야 하는지 혼란을 만들고,
서로 다른 API 호출 순서에 따라 설정 값이 덮어써질 위험이 있다.

## 2. 결정

상태별 소유 API를 하나로 고정한다.

- 식사 시간과 `먹지 않음` 여부는 `profile` 초기 입력과 `mypage/meal-times`가 소유한다.
- 운동 요일/시간 일정은 `profile` 초기 입력과 `mypage/exercise-schedules`가 소유한다.
- 식사/운동 기록 리마인더 수신 여부는 `recordReminderEnabled` 하나로 노출한다.
- 알림 설정 API는 수신 여부만 수정하고 식사 시간이나 운동 일정을 변경하지 않는다.
- 그룹 페이지의 그룹원 조회는 피드 진입에 필요한 표시 정보를 소유한다.
- 마이페이지의 그룹 구성원/나가기 경로는 계정 관리 성격으로 유지하되, 피드용 응답은 그룹 API를 우선한다.
## 3. 결과

긍정적 효과:

- 클라이언트가 각 상태를 어느 API에서 수정해야 하는지 명확해진다.
- 알림 on/off 변경이 식사 시간이나 운동 일정을 의도치 않게 덮어쓰지 않는다.
- 피드 화면에 필요한 미확인 기록 수는 그룹 API에 응집된다.

부정적 효과 / 리스크:

- 기존 `mealReminderEnabled`, `exerciseReminderEnabled` 요청/응답을 사용하던 클라이언트는 `recordReminderEnabled`로 변경해야 한다.
- DB에는 식사/운동 알림 플래그가 분리되어 있으므로 서버 내부에서는 같은 값으로 동기화해야 한다.
- 미확인 기록 수를 계산하려면 사용자의 그룹원 기록 상세 진입 시각을 별도 상태로 저장해야 한다.
