# ADR-0005: 소셜 로그인 전략 패턴 도입

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-06-29 |
| 관련 | LLD-0005 |

## 맥락 (Context)

소셜 로그인은 공급자마다 토큰 검증 방식, 사용자 정보 응답, 식별자 필드가 다르다.
이 차이를 `OAuthService` 안의 `if` 또는 `switch` 분기로 처리하면 다음 문제가 생긴다.

- 전략 선택, 외부 호출, 사용자 정보 파싱, 회원 처리, 토큰 발급 책임이 한 클래스에 섞인다.
- 신규 공급자 추가 시 기존 서비스 코드를 수정해야 하므로 OCP를 지키기 어렵다.
- 공급자별 오류 로그와 테스트 범위가 불명확해진다.
- 공통 회원 처리 로직과 공급자별 연동 로직이 반복된다.

템플릿 메서드 패턴도 검토했지만, 공유할 상위 상태나 훅 메서드가 크지 않고 상속 결합이 생긴다.
현재 구조에서는 인터페이스 기반 전략 패턴이 더 단순하다.

## 결정 (Decision)

소셜 로그인 공급자별 로직은 `OAuthStrategy` 구현체로 분리한다.
`OAuthService`는 전략 선택과 응답 조합만 담당하는 오케스트레이터로 둔다.

구성 기준은 다음과 같다.

- `OAuthStrategy`: 공급자별 사용자 정보 조회와 식별자 추출을 담당한다.
- `OAuthStrategyFactory`: `LoginType`에 맞는 전략을 반환한다.
- `OAuthMemberProcessor`: 공급자와 무관한 회원 조회/생성, 토큰 발급, 신규 상태 판단을 담당한다.
- 신규 공급자 추가 시 기존 서비스 코드는 수정하지 않고 전략 구현체와 매핑만 추가한다.

## 고려한 대안 (Considered Options)

1. **서비스 내부 분기 처리** — 초기 구현은 빠르지만 공급자가 늘수록 책임이 섞인다.
2. **템플릿 메서드 패턴** — 공통 골격이 강할 때 유리하지만 상속 기반 결합이 생긴다.
3. **전략 패턴** — 클래스 수는 늘지만 공급자별 변경 범위가 명확하고 테스트하기 쉽다.

## 결과 (Consequences)

### 긍정

- 공급자별 외부 연동 로직을 독립적으로 테스트할 수 있다.
- 회원 처리와 토큰 발급 공통 로직을 재사용한다.
- 신규 로그인 타입 추가 시 변경 범위를 전략 구현체로 제한한다.
- 장애 로그에서 어떤 공급자 호출이 실패했는지 추적하기 쉽다.

### 부정 / 트레이드오프

- 전략, 팩토리, 프로세서 등 클래스 수가 증가한다.
- 간접 호출 구조가 생기므로 패키지와 네이밍이 명확해야 한다.
- 공급자별 예외 코드를 공통 API 응답으로 변환하는 규칙을 유지해야 한다.

## 후속 / 미결정

- Apple 로그인은 운영 환경변수와 키 로테이션 정책을 별도로 검증한다.
- 공급자별 장애율과 응답 지연을 관찰할 수 있는 로깅/메트릭 보강을 검토한다.
