# ADR-0006: 테스트 시 Fake와 Mock 사용 전략

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-06-29 |
| 관련 | README, docs/lld |

## 맥락 (Context)

테스트에서 의존성을 어떻게 대체할지 일관된 기준이 필요하다.
Mock은 호출 횟수와 파라미터 검증에는 유용하지만 구현 세부사항에 테스트가 쉽게 묶인다.
Fake는 실제 인터페이스를 구현하므로 상태 기반 검증이 가능하지만 유지해야 할 코드가 생긴다.

특히 Spring Data JPA `JpaRepository`는 메서드가 많고 영속성 컨텍스트 동작까지 흉내내기 어렵다.
반면 OAuth 공급자 클라이언트, 스토리지 URL 발급기 같은 외부 연동 인터페이스는 API가 작고
테스트 목적에 맞게 Fake를 만들기 쉽다.

## 결정 (Decision)

테스트 더블은 의존성 성격에 따라 선택한다.

- `JpaRepository` 계열은 Fake를 만들지 않는다.
- JPA 저장소는 서비스 단위 테스트에서 Mock을 사용하거나, 필요 시 실제 DB 기반 테스트를 사용한다.
- 외부 연동 계층은 가능한 Fake를 우선 검토한다.
- 외부 연동 Fake가 지나치게 복잡해지면 Mock 또는 계약 테스트로 전환한다.
- 컨트롤러 문서 테스트는 MockMvc와 Mock 의존성으로 API 계약과 Swagger 예시를 고정한다.

## 고려한 대안 (Considered Options)

1. **모든 의존성을 Mock 처리** — 빠르지만 구현 세부사항 검증으로 흐를 위험이 크다.
2. **모든 의존성을 Fake 처리** — 상태 검증이 좋지만 JPA처럼 복잡한 의존성에는 비용이 과하다.
3. **의존성 성격별 선택** — 일관된 기준이 필요하지만 비용 대비 신뢰도를 조절할 수 있다.

## 결과 (Consequences)

### 긍정

- 테스트가 단순 호출 검증보다 실제 상태 변화를 더 잘 검증한다.
- JPA를 어설프게 재구현하는 비용을 피한다.
- 외부 호출 없이 OAuth, 스토리지, 알림 같은 연동 흐름을 안정적으로 테스트할 수 있다.
- Swagger 문서 테스트를 통해 API 응답 예시와 예외 케이스를 함께 관리한다.

### 부정 / 트레이드오프

- Fake 구현도 제품 코드 변경에 맞춰 관리해야 한다.
- 실제 외부 API와 Fake 사이에 차이가 생길 수 있다.
- DB 기반 테스트를 늘리면 실행 시간이 길어진다.

## 후속 / 미결정

- DB 통합 테스트가 늘어나는 시점에 Testcontainers 도입을 검토한다.
- 외부 연동 Fake는 실패 응답, 타임아웃, 잘못된 페이로드 케이스를 함께 제공한다.
