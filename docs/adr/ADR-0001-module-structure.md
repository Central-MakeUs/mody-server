# ADR-0001: 모놀리식으로 시작하고 추후 모듈 추출

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

## 맥락 (Context)

mody는 신규 CMC 백엔드 프로젝트다. 배포는 Trunk-Based Development(main 직머지 → 머지 시 자동 배포)로, 배포 산출물이 하나라고 가정한다. 향후 batch성 작업이 예상되지만 현재 규모는 작다.

도메인별 모듈과 API 모듈을 초기에 세분화하는 방식도 가능하지만, 신규 프로젝트 초기에는 빌드/구성 오버헤드가 크다. batch를 별도 부팅 앱(모듈)으로 분리하면 "main 머지 → 무엇을 배포?"가 두 갈래가 되어 TBD 단일 배포 흐름과 충돌한다.

## 결정 (Decision)

**모놀리식 단일 모듈로 시작한다.** 패키지는 feature 단위로 경계를 명확히 둔다 (`cmc.mody.<feature>`). batch가 독립 배포/스케일이 필요해지는 시점에 `core`(도메인·엔티티 공유) + `api` + `batch` 구조로 추출한다.

## 고려한 대안 (Considered Options)

1. **모놀리식 시작, 추후 추출** (채택) — 빠른 시작, TBD 단일 배포와 정합, 낮은 빌드 복잡도. 단, 나중 추출 비용과 패키지 규율 필요.
2. **core + api + batch 3모듈** — batch 독립 배포/스케일에 유리하나, 현재 단계엔 배포 파이프라인이 두 갈래가 되는 부담.
3. **도메인별 세분화** — 규모 확대가 확실할 때 유리하나 신규엔 과한 오버헤드.

## 결과 (Consequences)

### 긍정
- 초기 개발 속도가 빠르고 구조가 단순하다.
- TBD 단일 자동 배포와 자연스럽게 맞는다.
- 빌드/의존성 구성이 가볍다.

### 부정 / 트레이드오프
- 나중에 모듈로 추출할 때 리팩터링 비용이 든다.
- 모놀리식이라도 feature 패키지 경계를 스스로 지켜야 한다 (규율 의존).

## 후속 / 미결정
- [ ] 모듈 추출 트리거 기준 (언제 batch를 분리할 것인가).
- [ ] feature 패키지 경계 상세 규칙.
