# ADR-0014: 연속 기록 일수 계산 전략

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-04 |
| 관련 이슈 | #64 |

## 1. 맥락

날짜별 식사/운동 기록 조회 API는 작성자가 며칠 동안 연속으로 기록했는지 응답해야 한다.
연속 기록 일수는 피드에서 노출되는 표시 값이지만, 기록 생성/삭제, 그룹 탈퇴, 소프트 삭제 정책의 영향을 받는다.

따라서 단순 응답 필드 추가가 아니라 계산 시점과 저장 방식을 결정해야 한다.

## 2. 대안

### 대안 A: 조회 시점 계산

- 방식: 날짜별 기록 조회 시 작성자별 기록 날짜를 조회해 기준 날짜부터 역순으로 연속 일수를 계산한다.
- 장점: 별도 테이블과 배치가 필요 없고, 기록 삭제/그룹 나가기 같은 상태 변경이 즉시 반영된다.
- 단점: 피드 조회 시 작성자 수만큼 추가 쿼리가 발생할 수 있다.
- 적합한 상황: 초기 데이터 규모가 작고, 정확도와 구현 단순성이 더 중요한 단계.

### 대안 B: 일 배치 집계

- 방식: 매일 새벽 회원별/그룹별 streak 값을 별도 테이블에 계산해 저장한다.
- 장점: 조회 비용이 낮고 응답 시간이 안정적이다.
- 단점: 당일 기록 작성 직후 값이 늦게 반영될 수 있고, 삭제/탈퇴/기록 수정 이벤트에 대한 보정 배치가 필요하다.
- 적합한 상황: 데이터 규모가 커져 조회 비용이 문제되고, 하루 단위 지연이 제품상 허용되는 단계.

### 대안 C: 기록 작성/삭제 시 동기 갱신

- 방식: 기록 생성 또는 삭제 트랜잭션에서 streak 값을 즉시 갱신한다.
- 장점: 조회 비용이 낮고 값이 즉시 반영된다.
- 단점: 기록 생성/삭제 로직이 집계 상태에 강하게 결합되고, 과거 날짜 기록이나 소프트 삭제 보정이 복잡하다.
- 적합한 상황: 실시간 값이 필요하고, 기록 변경 이벤트가 명확히 통제되는 단계.

## 3. 결정

1차 구현은 **조회 시점 계산**을 채택한다.

- 현재는 데이터 규모가 작고, 별도 배치 운영보다 정확성과 단순성이 더 중요하다.
- 소프트 삭제와 그룹 탈퇴 정책이 즉시 반영되어야 하므로 materialized 값보다 원천 기록 기반 계산이 안전하다.
- 성능 병목이 확인되면 일 배치 집계 또는 동기 갱신 전략으로 전환한다.

## 4. 결과

긍정적 효과:

- 별도 집계 테이블 없이 빠르게 구현할 수 있다.
- 기록 삭제, 그룹 탈퇴, 소프트 삭제 상태가 즉시 반영된다.
- 배치 실패나 집계 값 불일치 문제가 없다.

부정적 효과 / 리스크:

- 피드 페이지에 포함된 작성자 수만큼 추가 조회가 발생할 수 있다.
- 기록량이 늘면 응답 시간이 증가할 수 있다.
- 성능 문제가 생기면 집계 테이블을 추가하는 마이그레이션이 필요하다.

## 5. 재검토 기준

- 날짜별 피드 조회 p95 응답 시간이 유의미하게 증가할 때.
- 한 그룹 내 기록량이 커져 작성자별 기록 날짜 조회가 병목이 될 때.
- 모바일 화면에서 streak 값의 하루 단위 지연이 허용된다는 제품 판단이 확정될 때.
