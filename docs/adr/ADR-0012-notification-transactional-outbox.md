# ADR-0012: 알림 발송에 트랜잭셔널 아웃박스 패턴 채택

## 상태

Accepted

## 날짜

2026-07-04

## 맥락

모디는 댓글, 챌린지, 식사/운동 리마인더 등 다양한 도메인 이벤트에서 알림을 생성한다.
알림은 앱 내 목록에 저장되어야 하고, 가능한 경우 FCM 푸시로도 발송되어야 한다.

도메인 트랜잭션 안에서 FCM을 직접 호출하면 다음 문제가 생긴다.

- FCM 장애나 지연이 핵심 도메인 트랜잭션을 실패시키거나 오래 점유한다.
- DB 커밋이 실패했는데 푸시는 이미 발송되는 불일치가 생길 수 있다.
- 푸시 발송 재시도, 실패 추적, 중복 방지 정책을 도메인 서비스마다 반복하게 된다.
- 여러 서버 인스턴스에서 동시에 발송할 때 같은 알림을 중복 처리할 수 있다.

따라서 알림 저장과 외부 푸시 발송을 분리하되,
알림 생성 자체는 원 도메인 트랜잭션과 함께 커밋되어야 한다.

## 결정

알림 발송은 **트랜잭셔널 아웃박스 패턴**으로 처리한다.

- 도메인 서비스는 직접 FCM을 호출하지 않고 알림 요청 이벤트를 발행한다.
- 알림 이벤트 리스너는 같은 트랜잭션 안에서 알림 row를 저장한다.
- `notification` row 자체가 사용자 알림이자 발송 outbox 역할을 한다.
- 커밋 이후 별도 스케줄러가 발송 대상 알림을 선점하고 비동기로 푸시를 발송한다.
- 푸시 구현은 `PushNotificationClient` 포트 뒤에 둔다.
- 1차 푸시 어댑터는 Firebase Admin SDK 기반 FCM으로 구현한다.
- FCM은 멀티캐스트 전송 제약에 맞춰 최대 500개 토큰 단위로 나누어 발송한다.
- 실패한 발송은 retry count, next retry time, last error를 저장하고 재시도한다.
- 읽음 상태는 발송 상태와 별도로 관리한다.

## 고려한 대안

1. **도메인 트랜잭션 안에서 FCM 즉시 호출**
   - 장점: 구현이 단순하고 지연이 작다.
   - 단점: 외부 API 장애가 핵심 트랜잭션에 전파되고, DB/푸시 불일치와 재시도 처리가 어렵다.

2. **알림 저장 후 동기 발송**
   - 장점: 알림 row는 남길 수 있다.
   - 단점: 발송 실패 처리가 요청 흐름에 남고, 대량 발송 시 API 응답 시간이 길어진다.

3. **별도 outbox 테이블 사용**
   - 장점: 발송 메시지와 사용자 알림 모델을 분리할 수 있다.
   - 단점: 현재 알림 도메인에서는 사용자 알림과 발송 단위가 거의 같아 중복 저장 비용이 크다.

4. **notification row를 outbox로 사용**
   - 장점: 저장/조회/발송 상태를 한 row에서 관리하고 초기 구현이 단순하다.
   - 단점: 알림 조회 모델에 발송 상태 필드가 함께 존재하므로 엔티티 책임이 커진다.

## 결과

### 긍정

- 도메인 트랜잭션은 알림 저장까지만 책임지고 외부 FCM 장애와 분리된다.
- DB 커밋에 성공한 알림만 발송 대상이 된다.
- 발송 실패, 재시도, 중복 방지, 선점 상태를 서버에서 추적할 수 있다.
- FCM 외 다른 푸시 provider가 필요해져도 포트/어댑터 교체로 확장할 수 있다.
- 알림 목록 조회와 푸시 발송이 같은 알림 row를 공유하므로 데이터 흐름이 단순하다.

### 부정 / 트레이드오프

- 스케줄러, 비동기 executor, 선점 쿼리, 재시도 상태가 필요해 구현 복잡도가 증가한다.
- 즉시 발송보다 수 초 단위 지연이 생길 수 있다.
- `notification` 엔티티가 사용자 읽음 상태와 발송 상태를 모두 가진다.
- 여러 인스턴스 운영 시 선점 쿼리와 인덱스 설계를 신중하게 관리해야 한다.

## 후속 결정

- 알림 배치와 스케줄러 운영 전략은 ADR-0018에서 다룬다.
- 야간 발송 제한 또는 quiet time 정책.
