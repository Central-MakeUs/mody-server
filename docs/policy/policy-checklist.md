이 문서는 API/엔티티 구현 전에 제품 정책 확인이 필요한 항목을 모은다. 결정된 내용은 ERD, LLD, ADR 중 성격에 맞는 문서에 반영한다.

## Auth / OAuth

- 기본 방향은 `LLD-0005`의 OAuth 전략 구조를 따른다.
- provider별 외부 API 차이는 전략 객체로 분리하고, 회원 생성/조회와 JWT 발급은 공통 흐름으로 처리한다.
- 회원 매칭 기준은 이메일이 아니라 `loginType + providerUserId`로 둔다.
- 로그인 성공 시 access token, refresh token, `isNewMember`를 응답한다.
- refresh token은 Redis가 아니라 DB에 저장하고, 재로그인 시 회원별 기존 token을 교체한다.
- 1차 출시 소셜 로그인 범위: 카카오, 애플, 구글을 모두 지원할지 확인한다.
- OAuth 필수 저장 값: provider user id는 필수로 두고, 이메일, 이름, 프로필 이미지 중 추가 저장 대상을 정한다.
- 이메일이 없는 OAuth 계정의 가입 가능 여부와 대체 식별 방식을 정한다.
- 탈퇴 후 같은 소셜 계정으로 재가입할 때 기존 데이터 복구 여부를 정한다.

## 회원 / 프로필

- 닉네임 중복은 허용한다. 닉네임은 그룹 안에서 표시되는 이름으로 보고 고유성 검증을 하지 않는다.
- 닉네임 변경 제한 정책을 정한다.
- 프로필 이미지는 소셜 프로필만 사용할지, 직접 업로드도 허용할지 정한다.
- 생년월일, 현재 체중, 목표 체중의 필수 입력 여부를 정한다.
- 건강 앱 연동 상태 저장 값과 변경 시점을 정한다.

## 그룹

- 회원당 활성 그룹 수 제한은 최대 4개로 갈지 최종 확인한다.
- 그룹 코드 만료, 재생성, 중복 방지 정책을 정한다.
- 그룹장 개념을 둘지, 없다면 빈 그룹 삭제 조건을 정한다.
- 그룹 탈퇴 후 해당 그룹 안에서 생성한 기록, 댓글 등 그룹 관련 정보는 삭제 처리하고 조회에서 제외한다.

## 기록 / 피드

- 식사/운동 기록 수정 및 삭제 가능 여부를 정한다.
- 기록 삭제 시 댓글과 이미지를 함께 숨길지 정한다.
- 이미지 필수 여부와 하루 기록 개수 제한 여부를 정한다.
- 식사 시간은 사용자 입력 시간과 업로드 시간 중 어떤 값을 기준으로 노출할지 정한다.
- 운동 시간 입력 단위를 분 단위로 고정할지, 시작/종료 시간으로 받을지 정한다.

## 댓글

- 댓글 수정 및 삭제 가능 여부를 정한다.
- 탈퇴한 회원의 댓글은 삭제 처리하고 조회에서 제외한다.
- 댓글 알림 대상이 기록 작성자만인지 같은 그룹원 전체인지 정한다.
- 댓글 최대 길이와 금칙어/신고 정책 필요 여부를 정한다.

## 체중

- 하루에 여러 체중 기록을 허용할지 정한다.
- 같은 날짜 재입력 시 새 기록을 추가할지 기존 기록을 수정할지 정한다.
- 증감 기준을 직전 기록 기준으로 할지 전날 기록 기준으로 할지 정한다.
- 탈퇴 후 체중 기록은 삭제 처리하고 조회에서 제외한다.

## 챌린지

- 주간 챌린지는 서비스 제공 목록에서 운영자가 선택하는지, 그룹/사용자가 선택하는지 정한다.
- 주간 챌린지 마감 기준을 요일만으로 표현할지 정확한 날짜/시간까지 관리할지 정한다.
- 사진 인증 수정/삭제 가능 여부를 정한다.
- 걸음수 챌린지 변경 가능 횟수와 변경 주기를 정한다.
- 걸음수 챌린지 변경 시 기존 기록의 보존, 순위 제외, 지역 내역 포함 범위를 정한다.
- 뱃지 지급 기준과 중복 지급 가능 여부를 정한다.

## 알림

- 댓글, 챌린지, 식사/운동 알림의 on/off 단위를 정한다.
- 식사/운동 알림 시간이 회원별 설정인지 그룹별 설정인지 정한다.
- 운동 알림을 요일별 낮/저녁 시간 모두 발송할지 정한다.
- 발송 실패 재시도 여부와 알림 보관 기간을 정한다.
- 개별 읽음 처리만 제공할지 전체 읽음 처리도 제공할지 정한다.

## 이미지 / 파일

- 이미지 저장소와 DB 저장 값이 URL인지 object key인지 정한다.
- Presigned URL 만료 시간, 파일 타입, 파일 크기 제한을 정한다.
- 업로드 후 실제 기록에 연결되지 않은 이미지 정리 정책을 정한다.
- 삭제된 기록/탈퇴 회원 이미지는 삭제 대상이다. 실제 객체 삭제 시점은 배치 또는 후속 정리 로직에서 다룬다.

## 삭제 / 보관

- soft delete 적용 대상 테이블 범위를 정한다.
- unique 충돌 처리는 `deleted_at` 포함 unique와 삭제 시 suffix 부여 중 선택한다.
- 탈퇴 회원의 개인정보, 피드, 댓글, 챌린지 데이터는 삭제 처리하고 조회에서 제외한다.
- 실제 DB 처리 방식은 물리 삭제와 soft delete 중 구현 시점에 결정하되, 서비스 응답에서는 모두 삭제된 것으로 취급한다.
- 법적 보관이 필요한 데이터가 있는지 확인한다.
