# issue

mody-server 작업을 시작하기 전에 GitHub Issue를 생성하고, 필요하면 `feature/{issue-number}` 브랜치를 만든다.

## 핵심 원칙

- 이슈 제목과 본문은 한글로 작성한다.
- 작업 범위가 크면 한 이슈에 몰아넣지 말고 사용자에게 분리안을 제안한다.
- 정책 미결정 사항은 결정된 것처럼 쓰지 말고 `확인 필요`로 남긴다.
- 이미 열린 관련 이슈가 있으면 새로 만들기 전에 재사용 여부를 확인한다.
- 브랜치는 사용자가 요청했거나 작업을 바로 진행할 때만 만든다.

## 절차

1. `gh issue list --state open --limit 30 --json number,title,labels`로 중복 이슈를 확인한다.
2. 요청 내용을 기준으로 이슈 제목을 정한다. 형식은 짧은 한글 명사구를 우선한다.
3. 아래 템플릿으로 본문을 작성한다.
4. `gh issue create --title "<title>" --body-file <tmpfile>`로 이슈를 만든다.
5. 라벨이 필요하면 기존 라벨을 먼저 확인한다: `gh label list --limit 100`.
6. 사용자가 브랜치 생성을 요청했거나 바로 작업을 이어가면 `git switch -c feature/<issue-number>`를 실행한다.
7. 생성한 이슈 번호, URL, 브랜치명을 사용자에게 보고한다.

## 이슈 본문 템플릿

```markdown
## 배경
<왜 필요한 작업인지>

## 작업 범위
- <구현/문서/테스트 단위>

## 완료 조건
- <사용자 또는 개발자가 확인할 수 있는 조건>

## 정책 확인 필요
- <있으면 작성, 없으면 "없음">
```

## 라벨 가이드

- 기능 구현: `feature`
- 버그 수정: `bug`
- 문서: `docs`
- CI/CD/배포: `ci`
- 리팩터링: `refactor`
- 논의/정책: `question`

라벨이 없고 생성 권한이 있으면 `gh label create`로 만든다. 권한이 없으면 라벨 없이 이슈를 생성하고 사용자에게 알린다.

## 하지 말 것

- 중복 이슈를 확인하지 않고 새 이슈 만들기.
- 한 이슈에 명세, 구현, 배포, 정책 결정을 모두 섞기.
- 사용자가 원하지 않았는데 main에서 바로 작업하기.
- 이슈 생성만 요청했는데 커밋이나 PR까지 진행하기.
